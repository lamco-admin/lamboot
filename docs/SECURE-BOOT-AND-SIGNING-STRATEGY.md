# LamBoot Secure Boot & Signing Strategy

**Updated:** 2026-04-05

---

## Current Status

LamBoot uses **self-signed keys + MOK enrollment** (Stage 1). The signing pipeline (`sign-lamboot.sh`) generates RSA-2048 keys, adds SBAT metadata, and signs all binaries with `sbsign`. Users enroll the key via `mokutil --import`.

This works but requires manual key enrollment on each machine/VM. This document outlines the path forward.

---

## The UEFI Secure Boot Trust Chain

```
PK (Platform Key) — root of trust, ONE per system
 └─ KEK (Key Exchange Key) — authorizes changes to db/dbx
     ├─ db (Allowed Signatures) — trusted certs that can load binaries
     │   ├─ Microsoft Windows Production PCA 2011
     │   ├─ Microsoft UEFI CA 2011/2023 (third-party)
     │   └─ OEM certificates
     └─ dbx (Forbidden Signatures) — revoked hashes/certs (VETO, checked FIRST)

SEPARATE (software layer, managed by shim):
 └─ MOK (Machine Owner Key) — extends db without modifying firmware keys
     └─ MOKX — extends dbx
```

**Key fact:** KEK is for key management, NOT for verifying boot binaries. db/dbx do the actual trust decisions. MOK extends db at the software level via shim.

## Signing Options for LamBoot

### Option 1: Self-Signed + MOK Enrollment (Current)

```
UEFI firmware → distro shim (Microsoft-signed) → LamBoot (MOK-verified) → kernel
```

- Generate keys locally, sign with `sbsign`, user enrolls via `mokutil --import`
- No external approval needed
- Manual enrollment per machine/VM
- **Best for:** Development, testing, self-managed environments

### Option 2: Pre-Enrolled Keys in OVMF (Proxmox/VM)

```
UEFI firmware (custom OVMF with LamBoot key in db) → LamBoot → kernel
```

- Build custom OVMF with LamBoot's signing key pre-enrolled in db
- Zero user interaction — LamBoot loads without MOK enrollment
- Requires distributing custom OVMF or enrolling keys in Proxmox templates
- **Best for:** Proxmox VM fleets, controlled environments

### Option 3: Custom Shim via rhboot/shim-review (Future)

```
UEFI firmware → LamBoot shim (Microsoft-signed) → LamBoot → kernel
```

- Build a custom shim with LamBoot's signing key embedded
- Submit to `rhboot/shim-review` on GitHub for peer review
- Reviewed shim gets signed by Microsoft
- **Requirements:**
  - Start from shim 16.1+ source tarball
  - Reproducible Docker build
  - SBAT metadata in all signed binaries
  - CVE compliance and security posture
  - Open-source bootloader chain
  - Convincing case for a new bootloader (rhboot team is cautious)
- **Timeline:** Weeks to months for review process
- **Best for:** Public distribution, distro packages

### Option 4: Microsoft UEFI CA Direct Signing (Not Recommended Now)

- Microsoft Partner Program membership
- EV code signing certificate
- Annual independent security audit (OCP SAFE Program)
- No GPL v3 code (Microsoft will not sign it)
- **Cost:** Partner fees + EV cert + annual audit
- **Best for:** Large commercial products only

### Recommended Staging

| Stage | When | Approach | Approval Needed |
|-------|------|----------|-----------------|
| **1. Development** | Now | Self-signed + MOK | None |
| **2. Proxmox fleets** | Near-term | Pre-enrolled OVMF keys | None |
| **3. Public distribution** | When stable | Custom shim via rhboot/shim-review | Peer review |
| **4. OEM/enterprise** | If needed | Microsoft direct signing | Microsoft + audit |

## Cryptographic Details

| Parameter | Value | Notes |
|-----------|-------|-------|
| **Algorithm** | RSA-2048 + SHA-256 | Standard. All firmware and shim versions support it. |
| **Key hierarchy** | PK, KEK, db (3 separate key pairs) | Generated by `sign-lamboot.sh` |
| **Key validity** | 10 years (3650 days) | Self-signed, can be regenerated |
| **SBAT section** | Required by shim 15.3+ | `dist/sbat.csv`, added via `objcopy` |
| **SBAT generation** | `lamboot,1` | Increment on security fixes |

**Do NOT use RSA-4096.** MOK enrollment freezes on many systems with 4096-bit keys. RSA-2048 is the industry standard for Secure Boot.

## SBAT (Secure Boot Advanced Targeting)

Every signed LamBoot binary includes a `.sbat` PE section:

```csv
sbat,1,SBAT Version,sbat,1,https://github.com/rhboot/shim/blob/main/SBAT.md
lamboot,1,Lamco Development,LamBoot,0.2.0,https://github.com/lamco-admin/lamboot-dev
```

When a security vulnerability is found and fixed:
1. Increment the generation number (`lamboot,1` → `lamboot,2`)
2. Distros update their `SbatLevel` UEFI variable
3. Old LamBoot binaries with lower generation are automatically rejected by shim
4. This replaces the need to add individual binary hashes to dbx

## Ubuntu GRUB Integration Notes

### What Ubuntu's GRUB Does

Ubuntu uses GRUB loaded via shim, with:
- `grub.cfg` generated by scripts in `/etc/grub.d/` (not BLS)
- Kernel install hooks in `/etc/kernel/postinst.d/` and `/etc/kernel/postrm.d/`
- Plymouth splash screen integration via `splash quiet` kernel params
- All GRUB modules compiled into a single signed monolithic binary (no runtime `insmod` under Secure Boot)
- `os-prober` for multi-OS detection

### What LamBoot Needs for Ubuntu Coexistence

1. **Kernel install hooks** — `/etc/kernel/postinst.d/zz-lamboot` and `/etc/kernel/postrm.d/zz-lamboot` that create/remove BLS entries when kernels are installed/removed. Ubuntu does NOT use `kernel-install(8)` by default, so both hook systems are needed.

2. **Don't touch grub.cfg** — when coexisting with GRUB, LamBoot should leave `/boot/grub/grub.cfg` alone. BLS entries are the parallel discovery mechanism.

3. **Plymouth compatibility** — LamBoot's UEFI stub boot path already preserves the framebuffer correctly. BLS entries should include `splash quiet` when appropriate.

4. **Secure Boot chain** — when loaded via Ubuntu's shim, LamBoot's signing key must be in MOK (not just in the firmware db). The key must NOT have the "Module-signing only" OID — that restricts to kernel module signing and would prevent bootloader loading.

### What systemd-boot and rEFInd Do

| Bootloader | Signing Method | Notes |
|------------|---------------|-------|
| **systemd-boot** | Distro shim + signed binary, or auto-enroll custom keys from ESP (since v252) | Debian Trixie ships signed systemd-boot |
| **rEFInd** | Ships pre-signed with own key + MOK enrollment | `refind-install --shim --localkeys` |
| **Sprout** | Secure Boot "beta" status | Rust UEFI bootloader, still developing SB support |

## Security Levels

### Secure Boot is All-or-Nothing

There is no "partial" Secure Boot. When enabled, ALL EFI binaries must be signed and verified. However, practical granularity comes from the chain — each link chooses its verification behavior:

- Firmware verifies shim (or LamBoot directly if key is in db)
- Shim verifies bootloader (via db + MOK)
- Bootloader verifies kernel (optional — LamBoot's `secure.rs` calls ShimLock Verify)

### LamBoot's Three Detection States

```rust
enum SecureBootState {
    Disabled,          // SB off or not supported
    ActiveWithShim,    // SB on, loaded via shim (ShimLock protocol available)
    ActiveDirect,      // SB on, loaded directly (self-signed in firmware db)
}
```

### TPM vs Secure Boot

| Feature | Secure Boot | TPM Measured Boot |
|---------|------------|-------------------|
| **Purpose** | PREVENT unauthorized code | RECORD what ran |
| **Enforcement** | Blocks unsigned binaries | Never blocks — advisory only |
| **Verification** | At load time (before execution) | After boot (remote attestation) |
| **LamBoot PCRs** | N/A | PCR 4 (kernel), PCR 5 (config), PCR 12 (cmdline) |

Both are complementary. LamBoot implements both.
